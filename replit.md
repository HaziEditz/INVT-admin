# BookaWaka Admin Dashboard

A full admin dashboard for the BookaWaka/360Taxi fleet management system, providing live data visualization, reporting, and management capabilities.

## Run & Operate

- **Run Server**: `node server.js` (listens on port 5000)
- **Environment Variables**:
    - `FIREBASE_DB_SECRET`: Firebase service account secret for server-side reads
    - `FIREBASE_SERVICE_ACCOUNT`: Service account JSON for Admin SDK
- **Login**: Authenticate via Firebase Email/Password at `OwnerLogin.aspx`.

## Stack

- **Frontend**: Bootstrap 3.4.1, UIkit, jQuery, Google Maps JS API, DataTables, Handlebars.js, Parsley.js
- **Backend**: Node.js (single-file server: `server.js`)
- **Database/Auth**: Firebase Realtime Database (`taxilatest`), Firebase Authentication
- **Build Tool**: None — SSR page functions rendered directly in Node.js

## Where things live

- `server.js`: Main Node.js server (~18,000+ lines), all routing and page generation.
- `taxitime.co.nz/owner/firebase-config.js`: Firebase project configuration.
- `taxitime.co.nz/owner/assets/`: Frontend assets (CSS, images, fonts, custom JS).
- `taxitime.co.nz/owner/JsScripts/`: Frontend JS (Zones.js, Masterjs.js, Tariffs.js).
- `DEV_REFERENCE.md`: Canonical Firebase paths, required fields, timezone rules, data gap log.

## Architecture decisions

- **SSR Frontend**: All `.aspx` pages dynamically generated by Node.js using shared layout functions.
- **Firebase Realtime DB as Source of Truth**: Drivers, jobs, bookings, customers, configurations.
- **Client-Side Aggregation for Reports**: Raw Firebase data fetched client-side, aggregated into DataTables with CSV export.
- **Dedicated Council Portal**: Separate `/council-portal` for restricted TM data access.
- **Multi-Company Management**: Single admin account can manage multiple companies via `ownerGroupId` validated server-side.

## Product

- **Fleet Management**: Live dashboard for drivers, active jobs, bookings, customer data.
- **Financial Management**: Subscription & billing, bank accounts, card commission, invoices, promos.
- **Operational Config**: GPS config, service links, app version management, vehicle types, zones, tariffs.
- **Reporting**: Closed jobs, shifts, cars, drivers, invoices, NZ Hours of Work compliance.
- **Total Mobility (TM)**: TM tariffs, council approvals, trip tracking, batch processing.
- **Business Account & ACC**: CRUD for business accounts, ACC clients with POs and billing.
- **Notification System**: Live notification feed, message sending.
- **Driver Management**: Add/edit drivers with SHA-256 password hashing, vehicle assignment (uppercase), trip history panel, duplicate ID enforcement.

## Gotchas

- **Timezone Handling**: Always use `window.COMPANY_TZ` (from `companySettings/{cid}/timezone`). Never rely on browser timezone. Pages must declare `var NZ_TZ = window.COMPANY_TZ || 'Pacific/Auckland';` in their JS scope.
- **Password Hashing**: Driver and dispatcher passwords are stored as SHA-256 hex (`passwordHash`). Server login verifies SHA-256 first, then base64 (legacy), then plaintext (very legacy). Never store `profile.password` plaintext.
- **Vehicle IDs Always Uppercase**: `vehicleId` and all `assignedVehicles` array elements written to `drivers/{cid}/{uid}` must be `.toUpperCase()`. Driver app expects uppercase (e.g., `"TAXI02"`).
- **Firebase Root Permissions**: Root-level shallow reads (`/.json?shallow=true`) may be blocked; system falls back gracefully.
- **Legacy Systems**: Original ASP.NET backend is defunct; all `DataManager/Data.aspx` calls fail.
- **DataTables Export**: Only `copy`, `csv`, `excel` buttons enabled — `print` and `pdf` removed.
- **Owner Group Switching**: `POST /api/switch-company` requires matching `ownerGroupId` for both companies.
- **Duplicate Driver Check**: `/api/check-driver-duplicate` checks email, phone, licenseNumber, taxiLicenseNumber, AND dispatcherId. Client must pass `dispId` query param.
- **Duration field name**: Firebase completedJobs may store duration as `durationLabel` — `flattenJoback` reads this first before falling back to `duration`/`durationMin`.
- **Job Command Engine**: Owner-panel writes to job state MUST go through `POST /api/job/command` (types: `CREATE`, `UPDATE`, `ASSIGN`, `REASSIGN`, `CANCEL`, `COMPLETE`, `CORRECT_FARE`). State machine validates transitions, version field is incremented atomically via Firebase ETag + `if-match` PUT, and every change appends an event to `jobEvents/{cid}/{bookingId}`. Driver/Dispatch apps may keep writing directly until they migrate.
- **Job Lifecycle Guard**: `/api/admin-write` rejects (HTTP 409) any write to `joback/*` or `jobs/*` that touches lifecycle fields (`Status/jobstatus/DriverId/VehicleId/completedAt/cancelledAt/fare/version` — case-insensitive). Emergency overrides: pass `{ bypassJobGuard: true }` in the body (logged to stdout).
- **Safe Cleanup (Prune)**: `/api/prune-stale-jobs` now only deletes truly orphaned slots (no BookingId AND no Status AND no DriverId). Legacy criteria (stale offers >4h, terminal-status records) are NO LONGER auto-deleted — those may contain revenue/audit data. Every cleanup batch emits a `SAFE_CLEANUP` event to `jobEvents/{cid}/cleanup`.
- **Reports Final-State Filter**: `reportsPage()` filters `flat` to terminal statuses (`completed`, `cancelled`, `declined`, `expired`, `paid`) only. Records with no real status (placeholders like `'—'`, `'-'`, `''` are normalized) are kept for legacy compatibility. `invoicecreate` / `invoicesent` keep all rows for fare correction.
